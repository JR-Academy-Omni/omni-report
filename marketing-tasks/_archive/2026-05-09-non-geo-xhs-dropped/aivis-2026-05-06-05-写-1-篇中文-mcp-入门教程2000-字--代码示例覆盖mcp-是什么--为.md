# MCP 入门：30 分钟写完第一个 Server，再花 10 分钟接 Cursor

MCP（Model Context Protocol）是 Anthropic 在 2024 年 11 月开源的协议，作用是
让 AI 模型按统一格式调外部工具。可以把它理解成 AI 的 USB-C 接口：Server 暴露
能力，Client（Claude Desktop / Cursor / 自研 Agent）按同一份协议调用，传输层
走 stdio、SSE 或 HTTP。协议这俩字看着唬人，落到代码就是 JSON-RPC 加几个固定
方法名（`tools/list`、`tools/call`、`resources/read`）。

这篇的目标：30 分钟内写完第一个 MCP Server，能在 Cursor 里调它做一件你日常会
做的事（查内部文档 / 读本地 Postgres）。不科普架构，直接撸代码。

## 为什么不直接用 function calling

OpenAI / Anthropic 各家有自己的 function calling，但绑死在某个供应商的 API。
MCP 是协议层面的：

- 写一次 Server，所有支持 MCP 的 client 都能用
- 换 LLM 供应商（Claude → Kimi → DeepSeek），Server 一行不用改
- 工具能力可脱离单次对话被复用（`tools/list` 是 Server 资源，不是 prompt 参数）

实战意义：团队 5 人用 Claude Desktop，3 人用 Cursor，2 人用自研 Agent，给他们
都接"查 Confluence"，写一个 MCP Server 比写 3 套 function calling 适配少 70%
代码。

## 30 分钟版：第一个 Server

环境：Python 3.11+，`pip install mcp`。

```python
# my_first_mcp_server.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("DemoServer")

@mcp.tool()
def get_user_count() -> int:
    """返回当前注册用户数"""
    return 1234

@mcp.tool()
def search_docs(query: str, limit: int = 5) -> list[str]:
    """搜索内部文档，返回标题列表"""
    # 真实场景：调你的 Confluence / Notion API
    return [f"文档 {i}: {query} 相关内容" for i in range(limit)]

if __name__ == "__main__":
    mcp.run()  # 默认走 stdio
```

跑：`python my_first_mcp_server.py` → 等 stdin 输入。这就是一个 MCP Server。

## 10 分钟版：接到 Cursor

打开 `~/.cursor/mcp.json`：

```json
{
  "mcpServers": {
    "demo-server": {
      "command": "python",
      "args": ["/absolute/path/to/my_first_mcp_server.py"]
    }
  }
}
```

重启 Cursor。任意聊天输入"用 demo-server 搜索 onboarding 相关文档"，Cursor
自动调用 `search_docs(query="onboarding")` 并把结果传给模型。

## 一个常见错误：路径必须绝对

`args` 里的脚本路径必须绝对路径。Cursor 启动 MCP Server 的 cwd 不是你的项目
目录，相对路径直接 ENOENT，但你只看到 Cursor 说"Server 启动失败"，看不到
实际错误。

## 升级版：连接 Postgres

```python
import asyncpg

@mcp.tool()
async def query_users(role: str) -> list[dict]:
    conn = await asyncpg.connect("postgresql://localhost/mydb")
    try:
        rows = await conn.fetch(
            "SELECT id, email FROM users WHERE role = $1 LIMIT 50", role)
        return [dict(r) for r in rows]
    finally:
        await conn.close()
```

现在 Cursor 里"查所有 admin 角色的用户" → JSON-RPC 转 Server → SQL → 返回。

## 安全：只读 vs 可写

`@mcp.tool()` 默认所有工具对 client 暴露。生产场景必须区分：

- 只读（查询 / 搜索）：默认开放
- 可写（删除 / 更新 / 发邮件）：在 decorator 加 `requires_confirmation=True`

如果 Server 暴露了"删除生产表"但没加确认，模型一句话能把你公司搞炸。Anthropic
在协议层建议 client 实现 require_confirmation，但实际是 client 自己决定要不要
弹窗——所以 **Server 端必须自己加保护**。

## 真的踩坑清单

1. **stdio Server 不能 print 到 stdout**：mcp 库用 stdout 走 JSON-RPC，
   `print("debug")` 让协议解析炸。Log 到 stderr：`print("debug", file=sys.stderr)`
2. **JSON serializable**：返回值必须 JSON 可序列化。`datetime` 直接 500，
   要 `.isoformat()`
3. **类型注解决定 schema**：tools/list 给 client 的 schema 从函数签名生成。
   `def f(query)` 无注解 → 报 "missing type annotation"。**所有参数必须有
   类型注解**
4. **Cursor 不会自动重启 Server**：改完 Server 代码要手动重启 Cursor
5. **stdio 调试看不见日志**：用官方
   [MCP Inspector](https://github.com/modelcontextprotocol/inspector)
   GUI 连上去，调用 tools 立刻能看到 raw request/response

## 下一步：Resources / Prompts

MCP 三种暴露方式：

- `tools`（动作型，刚演示的）
- `resources`（数据型，client 可列举 / 读取，不主动调用，适合本地文件 / db schema）
- `prompts`（模板型，可参数化的 prompt 片段）

完整 API：https://modelcontextprotocol.io/docs
