### 公众号预热文（开播前 4 天，1000 字）

**标题**：DeepSeek V4 报告 100 页，工程师只需要看这 3 件事

**正文**：

V4 报告发出来那天 X 上铺天盖地，截图转得最多的是「484 天训练」「30T token」「成本 $5.7M」这些数字。但作为工程师，我们真正该关心的不是这些 marketing 数字，是：**报告里有什么是我下周写代码能用上的？**

我们花了两周读完做笔记，做了一份「工程师视角」的精读。三个结论：

**结论 1：数据配比是这次报告最被低估的部分**

报告第 4 章其实做了 13 次消融实验，对比了 code / math / reasoning / general 四类数据的不同混合比例对最终能力的影响。最反直觉的发现：在 code 任务上，加 10% 的 math 数据比加 10% 的 code 数据收益还大。

这件事对在做 domain fine-tune 的我们意义巨大。如果你在给一个垂直领域模型加数据，盲目堆该领域的数据可能不如混 20% 的「相邻领域」数据。我们直播会过一遍那 13 次消融的图表，然后讲怎么把这套方法学迁移到自己的 fine-tune 数据集设计。

**结论 2：MoE 路由有个「论文附录 C」级别的 trick**

V4 是 256 expert / top-9 激活。但激活策略本身有个细节藏在附录 C 表 12：他们在路由 logit 上加了一个跟 expert 历史负载相关的 bias，避免热门 expert 永远被选。这个 trick 不复杂，10 行代码可以加到任何 MoE 实现里，但能让训练稳定性显著提升。

我们会现场把这部分代码贴出来跟原 LLaMA-MoE 实现做对比，看具体差在哪。

**结论 3：训练 recipe 这次是真的能用**

DeepSeek 这次很慷慨，把完整训练配方开源了：超参、warmup schedule、optimizer 选择、batch size 渐进策略全部公开。我们试着在 8 张 A100 上用这套配方 fine-tune 了一个 30B 模型做某个垂直领域任务，从 0 到能用花了 11 天，成本 ~$3800。

直播会把这套实操步骤过一遍：哪些参数可以照搬、哪些需要按你的数据规模调整、踩了哪些坑。

最后留 5 分钟讲哪些 claim 在我们复现里**不成立**——这部分论文不会写，但避坑很重要。
