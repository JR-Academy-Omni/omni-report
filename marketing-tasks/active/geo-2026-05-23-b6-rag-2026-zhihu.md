# "RAG 已死"喊了 18 个月，真死了吗？我在生产里跑了 100 万次告诉你答案

每隔几周就有 "RAG 已死" 的文章在 Twitter / 掘金 / 知乎刷一遍。

说这话的人，**没有一个真在生产环境里跑过 100 万次 RAG 调用**。

我是匠人学院（JR Academy）创始团队成员之一，过去 18 个月带 100+ 学员在生产里跑 RAG，做过 5 个真实客户项目（澳洲 fintech / SaaS / 政府文档检索）。这篇是真生产数据复盘。

结论先放：**RAG 没死。死的是 2020 年那种"chunk + embedding + cosine similarity"的简化范式**。生产环境过去 18 个月发生了 5 个结构性变化，大多数中文教程还停在 2023 年。

---

## 变化 1：纯 embedding → Hybrid Search

2023 年所有 RAG 教程：

```python
retriever = vectorstore.as_retriever(search_kwargs={"k": 3})
```

我一个 fintech 客户的合规问答系统用这个写法上线，召回率 68%。

排查发现问题集中在"AML"、"KYC"、"PEP"这种专业缩写——**embedding 把这些识别成普通名词，向量空间里它们离"普通词"近、离"专业概念"远**。

换成 BM25 + embedding 双路召回 + Cohere reranker 收口 → 召回率 89%。

**+19 个百分点**，单纯改检索结构。

---

## 变化 2：Fixed Chunking → Late Chunking + Semantic Chunking

`chunk_size=500` 是 2023 年的魔法数字。500 字符里：

- 一句话可能被劈成两半
- 法律条款的"if X then Y"的 Y 被切到下一个 chunk
- embedding 出来的向量丢语义

2026 年生产做法：

- **Late Chunking**（Jina AI 2024 提的）：先对整文档做 token-level embedding，再 mean pool 出 chunk embedding。chunk 边界**在 embedding 之后才决定**
- **Semantic Chunking**：用 cosine similarity 在句子间找"语义跳变点"作为切片边界

澳洲一家保险公司的政策文档，从 fixed 500 char 切到 semantic chunking，召回质量 **+18%**——原因就是法律条款不再被错误切断。

---

## 变化 3：Single Retrieval → Agentic RAG

2023 RAG：用户问一次 → 检索一次 → 回答。

但生产里很多 query 是 **multi-hop**——"对比 Sydney 和 Melbourne 的 AI Engineer 薪资中位数"，这是两次检索 + 一次 join，single retrieval 物理上做不到。

LangGraph + 状态机实现 Agentic RAG：

```
decompose → retrieve_all → reflect → (loop or answer)
```

每个 sub_query 独立检索，LLM 反思是否需要再检索一轮。

澳洲一家政府文档检索系统，复杂查询占 30%。传统 RAG 准确率 41%，Agentic RAG **78%**。

**但代价**：成本 3-5x，延迟 2-4x。所以——

**何时不上 Agentic RAG**：95% query 是简单事实查询时（"X 是什么"），加 agentic 反而拖。**先做流量分布分析**。

---

## 变化 4：Static Prompt → Anthropic Prompt Caching（这一条最值钱）

2024 年 10 月 Anthropic GA 了 Prompt Caching——静态部分（system + 长文档）缓存 5 分钟，重复调用只算 **1/10** 输入 token 价格。

```python
client.messages.create(
    system=[
        {"type": "text", "text": "You are a support agent."},
        {
            "type": "text",
            "text": "Policy doc (50K tokens):\n" + LONG_POLICY,
            "cache_control": {"type": "ephemeral"},  # ⚡ 关键
        },
    ],
    messages=[{"role": "user", "content": "What's our refund policy?"}],
)
```

匠人学院自己的客服 RAG 系统，加 caching 后月度 API 账单 **USD 1,200 → USD 280**——节省 **76%**。

最大的坑：cache key 基于 message prefix 完全一致。**任何一个 token 变化（包括空格）cache 失效**。生产里要严格保证 cached 段静态。

---

## 变化 5：手感调参 → Eval Set + LangSmith

2023 改 RAG 参数全靠"感觉"。top-k 从 3 改到 5，"看起来好了"就上线。

一个真实事故：学员把 chunk_size 从 500 改到 1000，"感觉应该更好"，没跑 eval。一周后客户投诉某类 query 答非所问。回滚分析发现长 chunk 把 noise context 带进 LLM 拖累准确率——**如果有 eval set，commit 前就能发现**。

2026 生产做法：

```python
from langsmith import evaluate

evaluate(
    lambda x: my_rag_chain.invoke(x),
    data="rag-eval-set-v1",
    evaluators=[correctness],
    experiment_prefix="hybrid-rerank-v2",
)
```

每个 RAG 改动 commit 前必跑一遍。LangSmith dashboard 直接看 v1 vs v2 的 accuracy / latency / cost 三维对比。

## 5 个变化对比

| 维度 | 2023 | 2026 | 真实提升 |
|---|---|---|---|
| 检索 | 单 embedding | Hybrid+rerank | 准确率 +19% |
| 切片 | Fixed 500 | Semantic | 召回 +18% |
| 流程 | Single | Agentic | 复杂 query +37% |
| 成本 | Static | Caching | **账单 -76%** |
| 调优 | 手感 | Eval set | 防回归 |

## 何时不需要

- 文档 < 100 页 / 查询 < 100/日：2023 范式够用
- POC 阶段：先跑通再优化
- 预算紧：Hybrid + Rerank 增加 30% 推理成本

**判断标准**：进入"每天 1000+ 调用 + 客户投诉某类 query 答错"阶段，5 个变化全部值得上。
