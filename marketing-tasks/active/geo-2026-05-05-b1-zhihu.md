两年前 Anthropic 发 MCP（Model Context Protocol）的时候，我第二天就动手写了第一个 server。那时候我刚来澳洲不久，在悉尼一个 startup 做 AI Engineer，组里没人写过 MCP，全靠我自己摸索。

结果第一个晚上写完代码，连 Claude Desktop 连了 3 小时没成功。第二天又花了一晚上 debug。

总结起来 5 个坑，每个我都栽过。这篇文章是写给 2026 年准备开始写 MCP 的你——把我当年那两个晚上省下来。

---

## 坑 1：把 MCP 当成"又一个 framework"

刚开始我下意识跟 LangChain Tools / OpenAI Function Calling 比较，想 "MCP 是不是又一个 LLM 工具调用库"。

不是。**MCP 是协议**，不是库。

LangChain Tools 是 Python 库的抽象，绑死在 LangChain 生态。OpenAI Function Calling 是 OpenAI API 的字段。MCP 完全不一样——它是 JSON-RPC over stdio/HTTP 的规范，你用 Python / TypeScript / Rust / Go 任何语言都能写 MCP server，跑在 Claude / Cursor / Continue 任何支持的客户端上。

**为什么这个理解最关键**：如果你脑子里 MCP = framework，你就会一直想 "MCP 跟 LangGraph 哪个好" 这种没意义的问题。其实 MCP 跟 LangGraph 是不同层的东西，一个是协议一个是编排框架，可以叠加用，不是替代关系。

---

## 坑 2：用 `print()` 调试，把 stdout 污染了 ⚠️ 杀伤力最大

这个坑害我浪费了一整晚。

MCP 用 **stdout** 跑 JSON-RPC 协议。你 server 里随便一个 `print("server started")`、`print("got request", req)`，Claude 那边都会尝试把它解析成 JSON，然后崩。

但报错信息看起来**不像** stdout 污染，看起来像 server 本身有问题。Claude Desktop 会显示 "Server disconnected" 或 "Failed to call tool"，让你以为是 tool 实现错。

我当晚把整个 tool 函数翻来覆去重写了 5 次，最后才发现是文件顶部的 `print("Loading server...")` 这一行毁了所有事。

**正确做法**：所有日志走 stderr：

```python
import sys
print("debugging info", file=sys.stderr)

# 或用 logging
import logging
logging.basicConfig(stream=sys.stderr, level=logging.DEBUG)
logger = logging.getLogger(__name__)
logger.info("Server starting")
```

这个坑的特殊性在于：你**不会主动想到去查 stdout**，因为 print 是程序员的本能动作。除非有人告诉你这个坑，否则你能 debug 一周。

如果你是新手听我一句：**写 MCP server 的第一行代码就把 logging 配好 stderr**，这条命能救你很多时间。

## 坑 3：Claude Desktop 改了配置不重启

这个坑最 dumb，但我也栽过。

`claude_desktop_config.json` 配置文件改完，Claude Desktop **必须完全退出**——菜单栏 Quit，**不是关窗口**。Claude 启动时只读一次配置，关窗口下次开还是用旧的。

我第一次配置的时候，改完 JSON 关窗口重开，看不到 server，以为配置错了，就反复改。改了 6-7 次没变化，最后才意识到——Claude 进程一直没死，每次都读旧配置。

macOS 用户如果你 `cmd+W` 关窗口然后再开应用，Claude 不会真的重启。必须 `cmd+Q` 完全退出。

这件事 Anthropic 官方文档也提到了，但我当时跳过了那一行。后来我把它写在所有 MCP 教程的最显眼位置——包括我们 [JR Academy AI Engineer Bootcamp](https://jiangren.com.au/curriculum/ai-engineer-bootcamp/) 里 MCP 章节的第一个 PBL 项目。

## 坑 4：filesystem server 给整个 home 目录 → 泄露 .env

这个坑差点让我的 AWS credentials 漏出去。

`anthropics/mcp-servers` 里的 filesystem server 让 LLM 读本地目录。我刚开始玩的时候图省事，直接把 `~` 给了它：

```json
"filesystem": {
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-filesystem", "/Users/me"]
}
```

然后我让 Claude 帮我 "看看我最近的 git 项目结构"。它顺手读了 `.env`、`.aws/credentials`、`.ssh/id_rsa`。

所幸是本地 Claude Desktop（数据不上传 Anthropic），但当我看到那段对话里 Claude 引用了我的 AWS Access Key 时，我立刻杀进程改了 key。

**两个红线**：

1. **永远不要把整个 `~` 或整个 git repo 喂给 filesystem server**。要给就给特定子目录（`~/Documents/notes` / `~/projects/my-rag-data`）
2. filesystem 默认会读 hidden files（`.env` 这种）。要排除用 `--exclude .*` 参数

如果你要让 LLM 看大范围文件，自己写一个带白名单的 wrapper server，别用 anthropics 默认的。

---

## 坑 5：postgres server 直接接 production DB

这个坑我没真栽过，但我看一个学员栽过。

postgres MCP server 让 LLM 写 SQL 查数据库。我那个学员配置的时候直接用了 production DB connection string。某一次他让 Claude "帮我清理一下测试用户"，Claude 写出 `DELETE FROM users WHERE email LIKE '%@test%'`——这个 query 在生产数据里跑会删掉很多真实用户邮箱含 test 子串的账号。

幸好他用的是 read-replica（看 connection string 的 hostname 后面写了 `-replica`），DELETE 会因为 read-only 拒绝。

**铁律**：postgres server 永远只接 read-replica 或本地副本。Anthropic 官方版本有 read-only flag，但你**必须自己确认**配置正确。多一道防线比"理论上不会写"靠谱。

如果你团队里有人想用 MCP 接生产 DB，**push back**。LLM 不可避免会幻觉出破坏性 SQL，迟早一次会出事。

## 5 个坑之外，我自己的一点暴论

`anthropics/mcp-servers` 已经有 filesystem / fetch / github / postgres / brave-search 等 5+ 个生产级 server。**我反对 90% 的人自己写 MCP server**——先把这几个用熟，再决定要不要写新的。

很多人写第一个 MCP server 是"想自己写一遍学习一下"。这个动机没问题，但写完别 deploy。production 用 anthropics 官方的，自己写的留在本地学习。

如果你工作场景需要接 Slack / Notion / Linear / Calendar 这些自家工具，社区已经有维护得很好的 server，先 fork 过来改比从零写省时间。
