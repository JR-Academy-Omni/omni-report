# 9 年前我在 GitHub 开了个 AI 资源 repo，刚翻出来吓到了

前几天清家里旧硬盘的时候，翻到一个 2017 年的 backup 文件夹。

里面有一行命令历史：

```
git push origin master --set-upstream
```

是当年第一次推送 `awesome-ai-cn` 仓库的 commit。

我点进 GitHub 去看那个 repo——长草到我自己都把 URL 记错了，搜了 3 次才找到。

进去之后 commit history 停在 2022 年 11 月，issue 全是没人回的状态，star 数还停在 4 位数。

我按下 `Ctrl+F` 搜了几个关键词：

> "ChatGPT" — 0 个匹配
> "LangChain" — 0 个匹配
> "Claude" — 0 个匹配
> "RAG" — 0 个匹配
> "MCP" — 0 个匹配
> "agent" — 2 个匹配，但点进去都是 2017 年 RPA 时代的链接

---

**整个 LLM 时代它一秒都没有参与。**

我自己 9 年前在维护的东西，错过了它本该见证的全部历史。这种感觉很特别。

---

## 我花了一周，把 287 条链接全点了一遍

数据：

> ✅ **还活着的：54 条（19%）**
> ⚠️ **重定向了的：23 条（8%）**
> ❌ **404 / 域名过期 / repo archived：210 条（73%）**

73% 死了。

而且剩下 19% 里大部分是 NVIDIA / Google / Stanford 这种永远不死但内容也基本不更新的官方页。**真正"活着且有用"的，大概只剩 27 条**。从 287 到 27，10:1 的衰减。

我做的另一件事是看维护历史。这个 repo 最后一次实质性更新是 2022 年 11 月——那个月 OpenAI 发布了 ChatGPT。**它在故事开始前的最后一个月停止了**。

---

## 为什么 awesome-* 列表大部分都会死

写文章批评维护者偷懒是最容易的，但这不是真问题。

真问题是**结构性的**：

> 1. **分类按"技术品类"组织**，但技术品类每 3-4 年就翻一遍。2017 年的 SVM / Random Forest 分类在 2026 年没人看。
>
> 2. **链接是 star-driven 的**，不是 use-driven。某个 repo 上了 GitHub Trending 大家都加进列表，但 6 个月后那个 repo 可能就没人维护了。
>
> 3. **没有"我用过 + 还在用"的过滤**。维护者把别人推荐的链接也加进来，本人没有亲自验证。
>
> 4. **更新机制依赖个人意愿**。个人意愿在第二年就消失了。

这 4 条加在一起，几乎注定一个"个人维护的 awesome-* 列表"撑不过 5 年。

---

## 我们在悉尼做了 4 年 AI 教学，学到了什么

我们是匠人学院（JR Academy），项目制 AI 工程实战平台，澳洲。过去 4 年带 100+ 学员从转行到拿澳洲本地 AI Engineer offer。

这 4 年里我们持续在用、持续在淘汰资源。每一期 cohort 我们都会复盘"这一期学员用得最多的免费资源是什么 / 用了反馈最差的是什么"。

> 这个 use-driven 过滤的过程，**就是 awesome-* 列表本该有但绝大多数都没有的核心机制**。

所以这次重启的不是 2017 年那个老 repo（它的结构就是错的）。重启的是一份基于过去 4 年真实使用数据精选的 **awesome-ai-cn 2026 版**。

## 4 条维护规则（避免再死一次）

新版的 GitHub repo 写进 README 的硬规则：

> 1. 每个条目必须有 `last-verified` 日期，季度自动跑 link checker
> 2. 每条目必须有"我们为什么用它"的一句话
> 3. 拒绝"上 Trending 自动加入"，新资源必须经至少一期 cohort 实际使用反馈
> 4. 不接受赞助，repo 永远 community-driven

这 4 条任何一条违反，列表会重新进入 9 年衰减的路径。

---

## 为什么这件事重要

> AI 资源列表的真实价值不在"链接多"，在"每个链接背后有人验证过、有人用过、有维护机制"。

9 年前的我没明白这件事，所以 287 条链接现在 210 条死了。

希望 2026 年版本走不同的路径。如果做对了，5 年后这份列表至少 70% 链接还活着（不是 27%）。如果做错了，5 年后再写一篇"翻出来吓到了"的文章吧。
